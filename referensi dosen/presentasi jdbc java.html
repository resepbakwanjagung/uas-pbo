BAB 1: Kebutuhan Data Persistence

📁 Mengapa Aplikasi Membutuhkan Database?

🧠 Studi Kasus: Sistem Manajemen Mahasiswa
Bayangkan Anda memasukkan 100 data mahasiswa baru ke dalam aplikasi. Tiba-tiba listrik padam atau aplikasi sengaja ditutup. Apa yang terjadi pada data tersebut jika hanya mengandalkan variabel lokal di dalam memori utama?

⚠️ Keterbatasan Penyimpanan RAM (ArrayList / Collection)

  • Volatile: Seluruh data langsung terhapus dari memori begitu proses Java Virtual Machine (JVM) berhenti.
  • Isolasi Data: Data internal aplikasi tidak dapat diakses atau dibagikan ke aplikasi lain secara aman.
  • Skalabilitas & Query: Melakukan penyaringan (filter) atau pengurutan (sorting) data dalam jumlah besar secara manual memakan resource komputasi yang tinggi.

📊 Komparasi Efisiensi: ArrayList vs Database Relasional

Kriteria Penilaian Struktur Data Ragam ArrayList (RAM) Sistem Database Relasional (Disk)
Sifat Persistensi Sifat sementara (Akan hilang saat program mati) Permanen tersimpan aman di dalam Media Harddisk/SSD
Kompleksitas Pencarian Linear Search berbiaya besar $O(n)$ Pencarian optimal berbasis terindeks struktur B-Tree / SQL
Integritas Aturan Tidak memiliki mekanisme validasi aturan baku Ketat (Mendukung validasi Primary Key & Foreign Key)
Akses Konkurensi Terbatas hanya untuk satu pengguna aktif aplikasi Mendukung Multi-user secara serempak (*Concurrent Access*)

BAB 2: Definisi & Komponen JDBC

🚗 Apa Itu JDBC?

📢 Analogi Transportasi Driver & Penumpang:
👨‍💻 Java Application: Bertindak sebagai Penumpang yang membawa barang (data) dan tahu tempat yang ingin dituju.
🛵 JDBC Driver: Bertindak sebagai Sopir Ojek/Taksi yang mengerti rute teknis jalan raya menuju lokasi tujuan.
🏢 Database Server: Lokasi Tujuan Akhir tempat data diturunkan, disimpan, atau dijemput kembali.

JDBC (Java Database Connectivity) adalah bagian dari spesifikasi standar Java SE berupa sekumpulan API (Application Programming Interface) yang menyediakan standardisasi interaksi aplikasi Java dengan mesin basis data relasional menggunakan bahasa query SQL.

graph LR App[Aplikasi Java] --> API[JDBC API] API --> Mgr[Driver Manager] Mgr --> Driver[Driver MySQL] Driver --> DB[(MySQL DB)]

🧩 4 Komponen Utama Abstraksi JDBC:

  • Driver: Pustaka pihak ketiga (*vendor-specific*) penghubung komunikasi protokol DB (ex: Connector/J).
  • Connection: Representasi sesi jalur komunikasi aktif tatap muka dengan database.
  • Statement: Wahana penampung eksekusi skrip SQL mentah menuju database.
  • ResultSet: Struktur data tabular hasil kembalian dari pembacaan perintah SELECT.
Abstraksi JDBC memisahkan kode Java dari ketergantungan tipe vendor DB tertentu Slide 3
BAB 3: Siklus Hidup Transaksi

⚙️ Arsitektur & Alur Kerja Standard JDBC

Setiap interaksi data dengan database melalui JDBC wajib mengikuti urutan siklus hidup (*lifecycle*) terstruktur berikut:

flowchart LR A[1. Load Driver] --> B[2. Buka Connection] B --> C[3. Create Statement] C --> D[4. Execute Query] D --> E[5. Fetch ResultSet] E --> F[6. Close Resource]
  • 1. Load Driver: Menginisialisasi kelas driver database ke memori agar siap digunakan oleh DriverManager.
  • 2. Buka Connection: Membuat jembatan koneksi fisik lewat port jaringan tertentu menggunakan kredensial URL, user, dan password.
  • 3. Create Statement: Membuat objek perintah SQL, baik yang bersifat statis maupun dinamis (*prepared*).
  • 4. Execute Query: Mengirimkan instruksi ke DB. Menggunakan executeUpdate() untuk modifikasi data, atau executeQuery() khusus pembacaan.
  • 5. Fetch ResultSet: Melakukan iterasi membaca baris data hasil seleksi dari database.
  • 6. Close Resource: Menutup semua gerbang objek koneksi untuk mencegah kebocoran memori (*memory leak*).
Rule Emas: Resource yang dibuka di langkah awal wajib ditutup di langkah akhir! Slide 4
BAB 4: Implementasi Dasar

🔌 Membuat Koneksi Database via Java

Spesifikasi Target: Nama Database: dbkampus | Struktur Tabel mahasiswa:

Nama KolomTipe DataAtribut Aturan
idINTPRIMARY KEY, AUTO_INCREMENT
nama / nim / jurusanVARCHAR(100) / VARCHAR(20) / VARCHAR(50)NOT NULL
 
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class KoneksiDemo {
    public static void main(String[] args) {
        String url = "jdbc:mysql://localhost:3306/dbkampus";
        String user = "root";
        String password = "";

        // Teknik Try-With-Resources: Otomatis menutup koneksi saat blok selesai
        try (Connection conn = DriverManager.getConnection(url, user, password)) {
            if (conn != null) {
                System.out.println("✅ Koneksi berhasil terhubung ke MySQL Server!");
            }
        } catch (SQLException e) {
            System.err.println("❌ Terjadi kegagalan koneksi: " + e.getMessage());
        }
    }
}
💡 Catatan Penting: Sejak spesifikasi JDBC versi 4.0+, pemanggilan eksplisit perintah lama seperti Class.forName() untuk memuat driver sudah tidak diwajibkan selama file JAR driver berada dalam *classpath* proyek.

BAB 5a: Operasi Menulis & Membaca Data

📝 Operasi CRUD – INSERT & SELECT

➕ Operasi INSERT (Create)

Menggunakan tanda tanya ? sebagai penampung parameter yang aman.

String sql = "INSERT INTO mahasiswa (nama, nim, jurusan) VALUES (?, ?, ?)";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setString(1, "Rian Sukarna");
    ps.setString(2, "202601050");
    ps.setString(3, "Informatika");
    
    int rowsAffected = ps.executeUpdate();
    System.out.println(rowsAffected + " data berhasil disimpan.");
}

🔍 Operasi SELECT (Read)

Menggunakan objek penampung ResultSet untuk membaca sekumpulan data tabel.

String sql = "SELECT * FROM mahasiswa";
try (Statement st = conn.createStatement(); 
     ResultSet rs = st.executeQuery(sql)) {
     
    while (rs.next()) {
        int id = rs.getInt("id");
        String nama = rs.getString("nama");
        String nim = rs.getString("nim");
        System.out.println(id + " | " + nama + " | " + nim);
    }
}
📌 Aturan Iterasi: Posisi pointer awal ResultSet berada tepat sebelum baris pertama data. Pemanggilan fungsi rs.next() akan menggeser pointer satu baris ke bawah sekaligus mengembalikan nilai Boolean false jika baris berikutnya kosong.
Gunakan executeUpdate() untuk mutasi data, dan executeQuery() khusus ekstraksi data Slide 6
BAB 5b: Operasi Mengubah & Menghapus Data

📝 Operasi CRUD – UPDATE & DELETE

✏️ Operasi UPDATE (Ubah Data)

Memperbarui isi kolom berdasarkan kriteria pengenal unik.

String sql = "UPDATE mahasiswa SET jurusan = ? WHERE id = ?";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setString(1, "Sistem Informasi");
    ps.setInt(2, 1);
    
    int rowsUpdated = ps.executeUpdate();
    System.out.println(rowsUpdated + " data berhasil diubah.");
}

❌ Operasi DELETE (Hapus Data)

Menghapus baris rekaman data secara permanen dari tabel.

String sql = "DELETE FROM mahasiswa WHERE id = ?";
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    ps.setInt(1, 3);
    
    int rowsDeleted = ps.executeUpdate();
    System.out.println(rowsDeleted + " data dihapus.");
}
⚠️ Peringatan Fatal Kegagalan Logika SQL:
Selalu pastikan klausa kondisional WHERE disertakan pada perintah UPDATE dan DELETE. Absennya klausa WHERE akan menyebabkan seluruh baris data pada tabel terhapus atau terubah secara tidak sengaja.
Fungsi executeUpdate() mengembalikan nilai integer berupa jumlah baris data yang terpengaruh Slide 7
BAB 6: Keamanan Celah Siber

🛡️ PreparedStatement vs Statement

⚠️ Celah Kerentanan SQL Injection

Menggabungkan input string secara langsung (*string concatenation*):

String inputId = "3 OR '1'='1'";
String sql = "SELECT * FROM user WHERE id = " + inputId;
// Hasil Kompilasi Query Akhir:
// SELECT * FROM user WHERE id = 3 OR '1'='1'

💥 Dampak Bahaya: Validasi logika jebol, penyerang siber bisa menguras seluruh isi database tanpa izin.

✅ Mekanisme Parameterized Binding

Menggunakan placeholder ? terproteksi otomatis:

String sql = "SELECT * FROM user WHERE id = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, inputId); 
// Input otomatis di-escape sebagai teks murni

✨ Keunggulan: Struktur query telah di-*compile* terlebih dahulu di awal oleh database, sehingga input berbahaya tidak akan dianggap sebagai perintah baru.

Aspek Perbandingan Objek Kelas Statement Biasa Objek Kelas PreparedStatement
Ketahanan SQL Injection Sangat Rentan / Bahaya Sangat Aman (Terproteksi)
Performa Eksekusi Berulang Lambat (Query harus di-parse ulang tiap kali jalan) Cepat (*Pre-compiled* & efisien untuk pengolahan batch)
Kerapian Struktur Kode Rumit, penuh tanda petik satu-dua bersilangan Sangat rapi menggunakan penanda urutan index parameterized
Standar Industri: Selalu gunakan PreparedStatement untuk menerima input pengguna Slide 8
BAB 7: Sinkronisasi Paradigma Objek Relasional

🧩 Menghubungkan Konsep JDBC dengan Pola OOP

Dunia OOP berpikir dalam bentuk **Objek & Referensi**, sedangkan database berpikir dalam bentuk **Baris & Kolom Tabel**. Menghubungkan keduanya membutuhkan pemisahan peran yang tegas.

1. Class Model / Entity (Representasi Objek)

Menerapkan prinsip Enkapsulasi penuh tanpa ada kode SQL.

public class Mahasiswa {
    private int id;
    private String nama, nim, jurusan;

    public Mahasiswa(String nama, String nim, String jurusan) {
        this.nama = nama; this.nim = nim; this.jurusan = jurusan;
    }
    // Sediakan standard Getter, Setter, dan Constructor...
}

2. Interface Abstraksi Akses Data

Kontrak baku blueprint fungsi tanpa memedulikan detail cara simpan.

import java.util.List;

public interface MahasiswaDAO {
    void insert(Mahasiswa m);
    List<Mahasiswa> findAll();
    void update(Mahasiswa m);
    void delete(int id);
}

BAB 8: Arsitektur Bersih Berbasis SOLID

🏗️ Penerapan Pola DAO (Data Access Object)

❌ Desain Antipattern (Buruk)

Mencampuradukkan logika bisnis dengan sintaks SQL di satu tempat:

public class Mahasiswa {
    private String nama;
    // Pelanggaran Berat Aturan SRP!
    public void simpanKeMySQL() {
        String query = "INSERT INTO mahasiswa...";
        // Kode JDBC tercampur di entitas objek
    }
}

✅ Desain Sesuai Standar DAO (Baik)

Entitas murni menampung data, urusan SQL diserahkan ke kelas DAO khusus:

// Service berinteraksi lewat abstraksi interface
public class MahasiswaService {
    private MahasiswaDAO dao; 
    
    public MahasiswaService(MahasiswaDAO dao) {
        this.dao = dao; // Loose Coupling (DIP Terpenuhi)
    }
}

💡 Korelasi Kuat Pola Desain dengan Prinsip SOLID:

  • Single Responsibility Principle (SRP): Class Mahasiswa hanya bertanggung jawab memodelkan entitas data mahasiswa. Sedangkan urusan mekanis CRUD ke database SQL menjadi tanggung jawab tunggal milik MahasiswaDAOImpl.
  • Dependency Inversion Principle (DIP): Komponen tingkat atas (*High-level service*) tidak boleh bergantung langsung secara kaku pada kelas implementasi database internal. Komponen harus bergantung pada abstraksi tingkatan `Interface` DAO.
Dengan DAO, migrasi dari MySQL ke PostgreSQL cukup mengganti implementasi tanpa menyentuh entitas Slide 10

BAB 9: Mini Project – Bagian 2

🎓 Bedah Kode Komponen Implementasi DAO

Berikut adalah cuplikan potongan kode inti implementasi JDBC murni di dalam tubuh kelas MahasiswaDAOImpl:

@Override
public void insert(Mahasiswa m) {
    String sql = "INSERT INTO mahasiswa (nama, nim, jurusan) VALUES (?, ?, ?)";
    // Otomatis membuka dan menutup objek Connection & PreparedStatement
    try (Connection conn = DatabaseConfig.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql)) {
        
        ps.setString(1, m.getNama());
        ps.setString(2, m.getNim());
        ps.setString(3, m.getJurusan());
        ps.executeUpdate();
    } catch (SQLException e) {
        throw new RuntimeException("Gagal melakukan operasi insert data mahasiswa", e);
    }
}

@Override
public List<Mahasiswa> findAll() {
    List<Mahasiswa> list = new ArrayList<>();
    String sql = "SELECT * FROM mahasiswa";
    try (Connection conn = DatabaseConfig.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql);
         ResultSet rs = ps.executeQuery()) {
        
        while (rs.next()) {
            Mahasiswa m = new Mahasiswa(rs.getString("nama"), rs.getString("nim"), rs.getString("jurusan"));
            m.setId(rs.getInt("id"));
            list.add(m);
        }
    } catch (SQLException e) {
        throw new RuntimeException("Gagal memuat daftar seluruh data mahasiswa", e);
    }
    return list;
}

BAB 10: Standar Rekayasa Perangkat Lunak

🏆 Panduan Best Practices Pengembangan JDBC

Untuk menghasilkan aplikasi berskala industri yang tangguh, aman, dan efisien, pastikan Anda selalu mematuhi standardisasi aturan penulisan kode berikut:

🔒 Aspek Keamanan & Kinerja

  • Gunakan PreparedStatement Selalu: Jangan pernah berkompromi menggunakan Statement biasa untuk nilai variabel dinamis yang berasal dari luar.
  • Terapkan Try-With-Resources: Menghilangkan risiko kelalaian lupa menutup resource yang bisa berujung pada hang-nya server basis data.
  • Batasi Ambil Data: Hindari kebiasaan buruk SELECT * jika tabel memiliki puluhan kolom; sebutkan nama kolom yang dibutuhkan secara spesifik.

🏗️ Aspek Manajemen Struktur

  • Sentralisasi Konfigurasi: Pisahkan parameter teks URL, User, dan Password ke dalam file eksternal .properties atau .env.
  • Gunakan Connection Pooling: Pada aplikasi web berskala besar, gunakan pustaka manajer kolam koneksi siap pakai seperti HikariCP demi efisiensi resource.
  • Isolasi Layer SQL: Jangan biarkan ada satu pun kata kunci SQL (seperti INSERT, SELECT) lolos bocor ke dalam kelas Model UI.
👉 Insight Penting: Pemisahan konfigurasi kredensial dari source code merupakan syarat mutlak agar kode aman saat diunggah ke repositori publik seperti GitHub.
Kualitas kebersihan kode menentukan tingkat kemudahan pemeliharaan aplikasi di masa mendatang Slide 13

Identitas Galat (Error) Akar Penyebab Utama Kelalaian Solusi Tepat Penyembuhan

B. Soal Analisis Studi Kasus Kode Program

Pertanyaan: Teliti bait kode di bawah. Identifikasi potensi bahaya kerusakan fatal apa yang mengancam keamanan aplikasi tersebut?

// Skenario pencarian data akun user internal aplikasi
String inputUser = request.getParameter("searchId");
Statement stmt = connection.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM mhs WHERE id = " + inputUser);

Kunci Analisis Jawaban Teknikal:
Kode di atas mengandung cacat arsitektur fatal berupa celah kerentanan keamanan **SQL Injection**. Solusi Perbaikan: Wajib segera dirombak menggunakan objek kelas PreparedStatement dan melakukan teknik binding argumen parameter ps.setInt(1, Integer.parseInt(inputUser)).
